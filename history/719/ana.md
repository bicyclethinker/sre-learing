# 二、哪些命令形成了有效重复

## 1. `ps` 的不同视角对比

下面这一系列不是机械重复：

```
ps -ef
ps aux
ps -efH
pstree -p
ps auxf
```

因为它们解决的问题不同：

- `ps -ef`：标准全进程列表
- `ps aux`：资源信息较丰富
- `ps -efH`：层次结构
- `pstree -p`：直观进程树
- `ps auxf`：BSD字段加树状关系

你实际上是在从不同角度观察同一个进程系统。

这是有效重复。

------

## 2. CPU、内存排序的纠错链

你先尝试：

```
ps -aux --sort %cpu
ps -aux --sort %MEM
ps aux --sort -%mem
```

最后逐渐接近了：

```
ps aux --sort=-%cpu | head
ps aux --sort=-%mem | head
```

这里虽然有不少语法错误，但目标始终明确：

> 找出 CPU 或内存消耗最高的进程。

属于有目标的纠错。

------

# 四、错误和错误习惯

## 1. 混淆 `kill` 和 `pkill`

你执行过：

```
kill -15 systemd
kill -9 sshd
kill ping
```

`kill` 后面通常需要的是 **PID 或 jobspec**，不是进程名称。

它的模型是：

```
kill 信号 PID
```

而按名称匹配的是另一类工具。

你其实已经会使用：

```
pkill ping
pkill -9 sshd
```

但还没有稳定区分两者。

今后输入前先问：

```
我手里的是 PID，还是进程名？
```

- PID：考虑 `kill`
- 名称：考虑 `pgrep` 后确认，再使用按名称操作的工具

------

## 2. 尝试终止 PID 1

你多次执行：

```
kill 1
kill -15 1
kill -9 1
```

PID 1 是系统初始化进程，当前系统中通常是 systemd。

这是非常危险的实验对象。

即使系统出于保护机制没有真正让它退出，也不应该把 PID 1 用作信号实验对象。

进程信号实验应使用自己创建的安全进程，例如：

- `sleep`
- `ping`
- 自己启动的测试脚本

不要使用：

- PID 1
- SSH 服务
- 网络管理服务
- 存储和数据库服务

------

## 3. 直接停止和强杀 sshd

你执行了：

```
systemctl stop sshd
pkill -9 sshd
```

如果你正通过 SSH 连接服务器，这可能导致：

- 当前连接断开
- 新连接无法建立
- 实验环境失联

服务管理实验最好使用专门安装的测试服务，例如你已经安装的 `httpd`，不要用维持当前远程连接的 `sshd` 做停止和强杀实验。

------

## 4. 管道两边的程序关系不清楚

例如：

```
ps -aux | ping
ping aux | grep ping
ps -aux | ping
ls | sudo
```

管道的含义是：

> 把左边程序的标准输出，作为右边程序的标准输入。

因此右边的程序必须能够合理接收左边的文本。

常见合理组合：

```
产生文本的命令 | 筛选或处理文本的命令
```

而 `ping` 并不是一个用于处理 `ps` 输出的文本过滤器。

你现在偶尔仍把 `|` 理解成“把两个相关命令连起来”，需要修正为：

> **左边输出的数据，右边是否真的会从标准输入读取？**

------

## 5. `ps -aux` 不是最推荐的写法

你大量使用：

```
ps -aux
```

更稳定清晰的是：

```
ps aux
```

或者：

```
ps -ef
```

`ps` 同时支持多套选项风格：

- Unix风格：带单横线
- BSD风格：不带横线
- GNU长选项：双横线

`aux` 属于 BSD 风格，通常不加前面的 `-`。

------

## 6. CPU、内存排序语法不稳定

你写过：

```
ps aux --sort -&mem
```

其中 `&` 在 Shell 中是控制符，可能把前面的命令放入后台，而不是表示 `%`。

这是一个典型的高风险拼写问题：

```
%mem
&mem
```

视觉上接近，但 Shell 含义完全不同。

复杂命令输入后，执行前应重新检查特殊字符：

```
&
|
>
<
%
;
```

------

## 7. 作业号和 PID 仍有混淆

你有时写：

```
fg 4
bg 4
fg 5
```

后来也正确使用：

```
fg %1
fg %vim
```

在 Bash 作业控制中，应重点区分：

| 标识          | 属于谁管理                 |
| ------------- | -------------------------- |
| PID           | Linux内核进程表            |
| `%1`          | 当前Bash的作业表           |
| `%vim`        | 当前Bash按命令名匹配的作业 |
| `jobs -p`输出 | 作业对应的进程PID          |

`jobs` 和 `fg/bg` 操作的是**当前 Shell 的作业**，不是系统上的所有进程。

这也是为什么：

> `ps` 中能看到的进程，不一定能通过当前终端的 `fg` 拉回来。

------

## 8. `nohup` 拼写与概念不稳定

你先执行：

```
nohub ping ...
```

正确命令是：

```
nohup
```

另外，`nohup` 本身不等于后台运行。

概念上应区分：

```
&       放入后台
nohup   忽略终端挂断信号
```

所以两者经常一起出现，但解决的是不同问题。

------

## 9. 把文件名当目录

你先执行：

```
mkdir 1.txt
```

然后：

```
touch 1.txt
```

即使名字以 `.txt` 结尾，它仍然是目录，因为 Linux 判断类型不看扩展名。

这次错误反而有学习价值，但以后看到类似问题应使用：

```
file 1.txt
ls -ld 1.txt
```

先确认类型，而不是继续猜。

------

## 10. 多个后台进程写同一个文件

你多次让不同 `ping` 进程重定向到：

```
2.txt
```

多个进程同时覆盖或写入同一个文件时，可能造成：

- 输出互相覆盖
- 内容混杂
- 后启动的重定向截断旧内容
- 难以判断输出属于哪个进程

每个实验进程最好使用独立文件，或者明确使用追加重定向。

------

## 11. 拼写错误仍然很多

例如：

```
httpf
oslinux
homenametcl
nohub
jons
ppstree
cim
```

今天的拼写错误比最初学习时少了一些，但在疲劳后明显增加。

这表明：

> 你的知识已经比输入准确度增长得更快。

现在不应继续追求命令数量，而要降低操作速度，尤其是包含 PID、信号和重定向符号的命令。

# 七、三个针对性练习

只给题目，不给答案。

## 练习一：基础复现——作业、PID与进程状态

### 背景

在当前 Bash 中启动三个安全的长时间任务，分别模拟：

- 前台任务
- 普通后台任务
- 输出重定向的后台任务

### 要求

1. 查看当前 Bash 的 PID。
2. 启动三个任务。
3. 记录每个任务的：
   - job编号
   - PID
   - PPID
   - STAT
   - COMMAND
4. 将一个前台任务暂停。
5. 把它放到后台继续运行。
6. 再把它拉回前台。
7. 使用正常终止信号结束一个任务。
8. 验证它是否真正退出。
9. 最后清理所有测试任务。

### 限制

- 不允许使用 `kill -9`。
- 不允许操作 PID 1。
- 不允许使用 sshd 作为实验对象。
- 每次状态变化后必须同时使用 Shell 作业工具和系统进程工具检查。

### 验收标准

能够解释：

```
job编号、PID、PPID、STAT、前台、后台、停止、运行
```

之间的关系。

## 练习二：综合场景——测试服务与后台监控

### 背景

服务器上运行一个专门用于实验的 Web 服务。你需要管理服务，并启动一个独立的后台监控任务。

### 要求

1. 确认测试服务是否已经安装。
2. 查看服务当前状态。
3. 启动服务。
4. 找出：
   - 服务主进程PID
   - 父进程
   - 完整命令行
   - CPU和内存使用情况
5. 重启服务，比较重启前后的PID。
6. 启动一个后台监控任务，定期把服务状态写入日志文件。
7. 让监控任务在退出当前终端后仍能继续运行。
8. 重新登录后找到该监控进程。
9. 停止监控进程。
10. 查看服务相关日志，确认操作时间和结果。

### 限制

- 不得使用 sshd 作为测试服务。

- 不允许让多个后台任务覆盖同一个日志文件。

```
systemd服务单元
服务主进程
Shell后台作业

```

## 练习三：故障排查——为什么 ps 能看到，jobs 却看不到

### 故障现象

管理员发现一个长期运行的测试进程：

- 使用系统进程工具可以看到。
- 当前终端执行 `jobs` 却没有显示。
- 使用 `fg %1` 提示没有对应作业。
- 该进程的父进程可能已经发生变化。
- 不确定它是普通后台任务、nohup任务，还是systemd服务。

### 排查要求

按顺序完成：

1. 确认当前 Shell PID。
2. 找到目标进程 PID。
3. 查看目标进程 PPID。
4. 查看完整进程树。
5. 检查当前 Shell 的作业表。
6. 判断目标进程是否由当前 Shell 创建。
7. 判断它是否仍与当前终端关联。
8. 判断它是否由 systemd 管理。
9. 写出为什么 `jobs` 看不到它。
10. 使用合适方式终止或管理该进程。

### 限制

- 不允许直接 `kill -9`。
- 不允许把“jobs没有显示”解释成“进程不存在”。
- 不允许用进程名盲目批量终止。
- 修改前必须先写出至少两个假设。
