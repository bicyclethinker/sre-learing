# 90分钟实验：公司项目目录权限管理

## 时间安排

| 实验                   | 预计时间 |
| ---------------------- | -------- |
| 一、用户和用户组初始化 | 15分钟   |
| 二、项目共享目录       | 25分钟   |
| 三、ACL精确授权        | 25分钟   |
| 四、故障排查与验收     | 25分钟   |

建议全程在专门的测试虚拟机中完成。

------

# 实验一：用户和用户组初始化

**预计15分钟**

## 背景

公司有开发组、测试组和审计组，需要创建对应的用户和用户组。

## 任务

创建以下用户组：

| 组名      | GID  |
| --------- | ---- |
| devteam   | 5100 |
| testteam  | 5200 |
| auditteam | 5300 |
| temporary | 5400 |

随后把：

```
temporary
```

改名为：

```
internteam
```

要求修改后 GID 仍然为 `5400`。

创建以下用户：

| 用户      | UID  | 主组     | 附加组    | Shell     |
| --------- | ---- | -------- | --------- | --------- |
| alice     | 6101 | devteam  | auditteam | 正常 Bash |
| bob       | 6102 | devteam  | 无        | 正常 Bash |
| carol     | 6201 | testteam | auditteam | 正常 Bash |
| backupsvc | 自动 | devteam  | 无        | 禁止登录  |

## 验收要求

你必须证明：

1. 四个用户组均已正确创建。
2. `internteam` 的 GID 仍为 `5400`。
3. 每个用户的 UID、主组、附加组和 Shell 正确。
4. `alice` 和 `carol` 都属于 `auditteam`。
5. `backupsvc` 存在，但不能获得正常交互式 Shell。
6. 普通用户的主目录真实存在。

## 限制

- 不得直接编辑 `/etc/passwd` 和 `/etc/group`。
- 查询单个用户时，不要直接输出整个 `/etc/passwd`。
- 修改附加组时，不能覆盖用户已有的附加组。
- 每次操作后必须立刻验证，不要全部做完再检查。

------

# 实验二：建立项目共享目录

**预计25分钟**

## 背景

开发组成员需要共同使用：

```
/srv/project
```

这个目录应由 root 管理，但开发组成员可以共同创建和修改文件。

## 任务

创建目录结构：

```
/srv/project/
├── src/
├── docs/
└── backup/
```

配置 `/srv/project` 及其子目录，使其满足：

1. 所有者为 `root`。
2. 所属组为 `devteam`。
3. `devteam` 成员可以进入、查看和创建文件。
4. 非 `devteam` 用户不能进入。
5. 项目成员创建的新文件和新目录自动继承 `devteam` 组。
6. 普通文件不应被无意义地赋予执行权限。

分别切换到 `alice` 和 `bob`：

- `alice` 在 `src` 中创建 `main.txt`。
- `bob` 在 `docs` 中创建 `readme.md`。
- 检查两个文件的所有者和所属组。

再切换到 `carol`：

- 尝试进入 `/srv/project`。
- 记录实际结果。

## 验收要求

你必须展示：

- `/srv/project` 目录本身的详细权限。
- `main.txt` 和 `readme.md` 的所有者、所属组和权限。
- 新文件是否继承 `devteam`。
- `carol` 能否访问及原因。
- 目录权限中的特殊位显示在什么位置。

## 限制

- 不允许使用 `chmod 777`。
- 不允许使用 `chmod -R 2777`。
- 查看目录自身时必须使用能区分“目录自身”和“目录内容”的查看方式。
- 不能只用 root 身份验证权限。

# 实验三：使用 ACL 精确授权

**预计25分钟**

## 背景

测试人员 `carol` 不属于开发组，但需要：

- 可以进入项目目录；
- 可以读取 `docs/readme.md`；
- 不能修改 `src/main.txt`；
- 将来在 `docs` 中创建的新文件，她也应该自动拥有读取权限。

审计人员需要能够读取项目文件，但不能修改。

## 任务A：单独给用户授权

在不改变 `carol` 主组和附加组的情况下，使其：

1. 可以穿过 `/srv/project`。
2. 可以进入 `/srv/project/docs`。
3. 可以读取 `readme.md`。
4. 不能修改 `readme.md`。
5. 不能进入 `src`，或者进入后不能读取 `main.txt`，由你选择一种设计。

每次修改 ACL 后，都要查看 ACL 内容。

## 任务B：默认 ACL

为 `/srv/project/docs` 设置默认 ACL：

- 将来新建的普通文件，`carol` 默认具有读取权限。
- 将来新建的子目录，`carol` 具有进入和读取目录内容的必要权限。
- 已经存在的 `readme.md` 不应因为设置默认 ACL 而自动发生变化。

设置后由 `alice` 在 `docs` 中创建：

```
plan.txt
release/
```

再检查新文件和新目录的 ACL。

## 任务C：ACL mask 实验

人为把 `/srv/project/docs` 的 ACL mask 缩小，使 `carol` 原本被授予的部分权限失效。

要求你观察：

1. ACL 条目中给 `carol` 写了什么权限。
2. `effective` 实际生效权限是什么。
3. 修改 mask 后实际访问是否发生变化。
4. 恢复合理的 mask。

## 验收要求

你需要解释：

- 普通 ACL 与默认 ACL 的区别。
- 默认 ACL 为什么只适合设置在目录上。
- 用户 ACL 条目存在时，为什么实际权限仍可能受 mask 限制。
- 删除某个指定用户 ACL 与删除全部 ACL 有什么区别。

## 限制

- 不允许通过修改基础权限为 `777` 解决。
- 不允许把 `carol` 加进 `devteam`。
- 不允许一上来清空全部 ACL。
- 测试权限时必须真正切换到对应用户。

# 实验四：故障排查

**预计25分钟**

先由 root 创建以下故障场景。创建时可以查看帮助，但修复阶段必须先排查再操作。

## 故障背景

用户 `intern01` 报告：

> 我可以登录服务器，但是进不了项目文档目录；管理员说已经给我配置了权限。

同时管理员发现：

- `intern01` 的用户组关系可能不正确；
- 目录上看起来有 ACL，但用户实际仍没有权限；
- 某个父目录可能缺少必要权限；
- ACL mask 可能限制了有效权限；
- 项目目录的所有者或所属组可能被设置错误；
- 一位管理员曾经使用过 `chmod 000`；
- 另一个管理员曾经使用过 `chmod 7777`。

## 最终目标

让 `intern01` 满足：

1. 可以正常登录并进入自己的主目录。
2. 可以穿过 `/srv/project`。
3. 可以读取 `/srv/project/docs/plan.txt`。
4. 不能修改 `plan.txt`。
5. 不能删除其他用户的文件。
6. 不属于 `devteam`。

## 强制排查顺序

### 第一步：检查用户

检查：

- 用户是否存在；
- UID；
- 主组；
- 附加组；
- 主目录；
- Shell。

### 第二步：检查完整路径

不能只检查目标文件。

必须依次检查：

```
/
└── srv
    └── project
        └── docs
            └── plan.txt
```

判断每一级目录是否允许该用户穿过。

### 第三步：检查基础权限

检查：

- 所有者；
- 所属组；
- user/group/other 权限；
- 是否存在异常特殊权限。

### 第四步：检查 ACL

检查：

- 是否有 `intern01` 的 ACL 条目；
- ACL mask；
- 默认 ACL 与当前文件 ACL；
- 实际有效权限。

### 第五步：提出假设

在修改之前，至少写出两个故障假设，例如：

```
假设1：
证据：

假设2：
证据：
```

### 第六步：逐项修复

每次只修改一个问题，然后立刻以 `intern01` 身份验证。

## 限制

- 不允许使用 `chmod 777` 或 `chmod 7777` 作为修复方法。
- 不允许把 `intern01` 加入 `devteam`。
- 不允许直接删除后重建用户。
- 不允许一开始就执行 `setfacl -b`。
- 不允许只看 `ls -l` 就下结论。
- 不允许使用 root 身份创建一次文件后就宣布成功。

## 练习重点

这套题不是考你能不能碰巧把权限调通，而是训练下面这个完整闭环：

```
创建用户和组
→ 设置文件所有权
→ 配置基础权限
→ 配置特殊权限
→ 使用ACL补充例外规则
→ 切换真实用户验证
→ 根据报错逐层排查
```

不要追求命令条数。90分钟内把四个场景做完整，比继续机械重复几百条命令更有效。
